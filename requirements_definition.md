# ファイル分類ツール — 要件定義書

> **本書の位置づけ**: 本書はファイル分類ツールを新規開発するための要件定義書です。
> 実装エージェントへの指示書として使用します。

---

## 1. プロジェクト概要

### 1.1. 目的

ファイル名に含まれる `[タグ名]` パターンに基づいてファイルを自動分類するデスクトップGUIツールを開発する。
日本語の作者名・サークル名での分類を主な用途とする。

### 1.2. アプリケーション名

| 項目 | 値 |
|---|---|
| 表示名（ウィンドウタイトル等） | ファイル分類ツール |
| EXEファイル名 | `FileClassifier.exe` |

### 1.3. 対象ユーザー

大量のファイルを作者名・サークル名ベースで整理する個人ユーザー。

### 1.4. 技術スタック

| 項目 | 採用技術 |
|---|---|
| フレームワーク | Tauri v2 |
| バックエンド | Rust |
| フロントエンド | React + TypeScript |
| UIライブラリ | Tailwind CSS + shadcn/ui |
| テーブル | TanStack Table（仮想スクロール・インライン編集対応） |
| 配布形態 | 単一EXE（WebView2 はWindows 10/11 標準搭載を前提） |
| データ形式 | JSON（設定ファイル・辞書ファイル） |

### 1.5. 動作環境

| 項目 | 要件 |
|---|---|
| OS | Windows 10 (21H2以降) / Windows 11 |
| ランタイム依存 | WebView2（OS標準搭載。未導入時はダウンロードブートストラップで自動インストール） |
| 高DPI対応 | あり（Tauri/WebView2 が標準対応） |

## 2. 機能要件一覧

| ID | 機能名 | 概要 |
|---|---|---|
| F-01 | ファイル分類 | 入力フォルダ内のファイルを辞書に基づいて出力フォルダへ振り分け |
| F-02 | 辞書作成（フォルダから） | 選択したフォルダのサブフォルダ構成から辞書を自動生成 |

| F-04 | 出力フォルダ整理 | 類似フォルダの検出・統合、および少数ファイルフォルダの検出・グルーピングを行う |

## 3. 機能要件 詳細

### F-01: ファイル分類

#### 3.1.1. 分類フロー

```
プレビュー編集による辞書更新（編集がある場合、→ セクション3.5.5 参照）
  ↓
入力フォルダ内のファイルを列挙（再帰スキャン対応、セクション3.1.4参照）
  ↓
各ファイルについて:
  1. プレビュー編集による分類先上書きを確認（→ セクション3.5.4 参照）
     - 上書きあり → 編集後の分類先を使用（ステップ2-4をスキップ）
  2. ファイル名から [タグ名] を抽出（→ セクション4「タグ名抽出」参照）
     - タグなしファイル → スキップ
  3. タグ名を正規化して分類キーを生成（→ セクション5「正規化」参照）
  4. 辞書で分類先フォルダ名を解決（→ セクション6「辞書」参照）
     - 辞書完全一致 → そのフォルダ
     - 類似マッチ → 一致フォルダ（辞書に正式登録）
     - どちらも不一致 → タグ名をフォルダ名として自動作成・辞書登録
  5. ファイル名補正オプションを適用（→ 3.1.3 参照）
  6. 出力先フォルダを作成（存在しなければ）
  7. ファイルをコピー or 移動
     - 同名衝突 → _1, _2, ... サフィックス付与（→ セクション10 参照）
  ↓
処理完了 → 辞書を保存、ログを記録
```

#### 3.1.2. ファイル操作モード

| モード | 動作 | デフォルト |
|---|---|---|
| コピー | ファイルをコピー（メタデータ保持） | ○ |
| 移動 | ファイルを移動 | |

#### 3.1.3. ファイル名補正オプション

分類先へのコピー/移動時に、ファイル名を以下のルールで補正する。各オプションは独立してON/OFF可能。

| オプション | 説明 | デフォルト |
|---|---|---|
| 著者名削除 | ファイル名から `[タグ]` 部分（角括弧含む）を削除する。削除後の連続空白は1つに圧縮し、前後の空白をトリムする | OFF |
| 数字を半角に統一 | 全角数字を半角に変換（NFKC正規化）し、数字前後の空白を除去する | OFF |

**適用順序**: 著者名削除 → 数字正規化

#### 3.1.4. スキャン範囲

入力フォルダのスキャンは以下の2モードをオプションで切り替え可能とする。

| モード | 動作 | デフォルト |
|---|---|---|
| 直下のみ | 入力フォルダ直下のファイルのみを対象とする | ○ |
| 再帰スキャン | 入力フォルダ配下のすべてのサブフォルダを再帰的にスキャンし、見つかった全ファイルを対象とする | |

再帰スキャン時の注意:
- サブフォルダの階層構造は分類先に反映しない（すべてフラットに分類される）
- 出力フォルダが入力フォルダの配下にある場合、出力フォルダ自体はスキャン対象から除外する

入出力フォルダ同一時の動作:
- 入力フォルダと出力フォルダに同一のフォルダを指定することを許可する
- この場合、再帰スキャンは強制的に無効化される（UIでグレーアウト表示し変更不可）
- ユースケース: フォルダ直下に追加したファイルをサブフォルダに分類する

#### 3.1.5. 実行方式

- ファイル操作はバックエンド（Rust）側で非同期実行し、UIをブロックしない
- 進捗をプログレスバーとステータスラベルでリアルタイム表示する
- フロントエンドへの進捗通知は Tauri のイベントシステムを使用する

### F-02: 辞書作成（フォルダから）

ユーザーが選択したフォルダ内の全サブフォルダをスキャンし、辞書を再構築する。

- 辞書編集画面のツールバーから「辞書作成」ボタンを押下し、フォルダ選択ダイアログを表示する。デフォルト選択は出力フォルダとする
- フォルダ選択後、インラインの確認バーを表示し、既存辞書の上書きについてユーザーの了承を得る
- 辞書作成は以下の2パスで行う:
  - **パス1（フォルダ名）**: 選択フォルダ内の各サブフォルダ名について、グループ化ルール（セクション6.2）に従いキーを登録する
  - **パス2（ファイル名）**: 各サブフォルダ内のファイル名から `[タグ]` を抽出し、`normalize(タグ)` → そのフォルダ名のマッピングを辞書に追加する。パス1で登録済みのキーは上書きしない（フォルダ名優先・先勝ち）
- 作成中は「辞書を作成中...」を表示する
- 生成完了後、辞書ファイルを保存する
- 完了後、ドライランの自動再実行をトリガーする（F-05参照）

---

### F-03: 辞書編集

辞書の内容をGUI上で操作する画面。辞書データ（`正規化キー → フォルダ名`）を**出力フォルダ単位でグルーピングして表示**し、直感的に編集できるようにする。

#### 3.3.1. 表示構造

辞書の内容を出力フォルダ（値）ごとにグループ化して表示する。各グループ配下にそのフォルダに紐づく分類キー（キー）を一覧表示する。

```
▼ 関西漁業協同組合 (丸新)
    関西漁業協同組合(丸新)
    関西漁業協同組合
    丸新

▼ むらの・たみ
    むらのたみ

▼ Ink Complex (智弘カイ)
    inkcomplex(智弘カイ)
    inkcomplex
    智弘カイ
```

#### 3.3.2. 操作

| 操作 | 説明 |
|---|---|
| 出力フォルダの新規追加 | 新しい出力フォルダ名を入力してグループを作成する。グループ化ルール（セクション6.2）に従い、フォルダ名から自動的に分類キーも生成・登録する |
| 分類キーの追加 | 既存の出力フォルダグループに分類キーを追加する。入力されたキーは自動正規化される |
| 出力フォルダ名の編集 | グループのフォルダ名をインラインで編集できる。変更するとそのグループ配下の全キーのマッピング先が更新される |
| 分類キーの編集 | 分類キーをインラインで編集できる。編集時に自動正規化される |
| 分類キーの削除 | 選択した分類キーを削除する |
| 出力フォルダの削除 | グループごと削除する（配下の全キーも削除される。確認ダイアログを表示） |
| 保存 | 辞書をファイルに保存し、ドライランの自動再実行をトリガーする（F-05参照） |

#### 3.3.3. バリデーション

- 分類キーは入力時に自動正規化される（セクション5のルール）
- 正規化後のキーが他の出力フォルダに既に登録されている場合はエラーを表示し、追加を拒否する
- 出力フォルダ名が空の場合はエラーを表示する
- 同一の出力フォルダ名が既に存在する場合はエラーを表示する

### F-04: 出力フォルダ整理

出力フォルダ内のサブフォルダに対して、以下の2つの整理機能を提供する。

| 機能 | 概要 |
|---|---|
| 類似フォルダを統合する | 類似名のフォルダ＋辞書の正式名を検出・統合 |
| ファイル数の少ないフォルダを整理する | ファイル数が少ないフォルダを検出し、ユーザー定義のグループに統合 |

右ペインの「出力フォルダ整理」ボタンからモーダルダイアログを開き、ダイアログ内のラジオボタンで機能を切り替える。
両機能とも統合処理後に辞書を自動更新し、ドライランの再実行をトリガーする。

---

#### 3.4A. 類似フォルダ統合

##### 3.4A.1. 類似判定の対象

以下の2種類の名前を合わせて類似判定の入力とする。

| 種類 | ソース | 説明 |
|---|---|---|
| 実フォルダ | 出力フォルダ内のサブフォルダ | 実際にディスク上に存在するフォルダ |
| 辞書名 | 辞書の値（分類先フォルダ名） | 辞書に登録されている正式フォルダ名。実フォルダとして存在しない場合もある |

辞書名は、実フォルダのいずれかと類似する場合のみグループに含める（実フォルダと無関係な辞書名は対象外）。

##### 3.4A.2. 類似判定用の正規化

類似判定では、通常の正規化（セクション5）に加えて**追加の正規化処理**を施した比較用キーを生成する。
これにより、記号の有無、全角半角の違い、カナ表記の違い、数字表記の違いなど、表記ゆれのあるフォルダ名を類似として検出できる。

**類似判定用の処理ステップ**（通常の正規化に続けて順序厳守で実施）:

| Step | 処理 | 説明 |
|---|---|---|
| 1 | 通常の正規化（セクション5） | NFKC正規化（全角英数→半角、半角カナ→全角カナ）、小文字化、空白除去、文字フィルタリング |
| 2 | 丸括弧・カンマの除去 | `(`, `)`, `,` を削除する |
| 3 | カタカナ→ひらがな変換 | 全角カタカナをひらがなに統一する（例: `ア` → `あ`） |
| 4 | 漢数字→アラビア数字変換 | `一`→`1`, `二`→`2`, ... `九`→`9`, `十`→`10`, `百`→`100`, `千`→`1000`, `万`→`10000` に変換する |

※ Step 1 の NFKC 正規化により、全角英数字→半角、半角カタカナ→全角カタカナは既に統一されている。
Step 3 でさらにカタカナ→ひらがなに統一することで、カナ表記の違いもすべて吸収する。

**結果**: 類似判定用キーは `isAlphanumeric` 判定が真の文字（英数字、ひらがな、漢字等）のみで構成される。

**例**:

| フォルダ名 | 通常の正規化キー | 類似判定用キー | 類似として一致するもの |
|---|---|---|---|
| `A・B` | `ab` | `ab` | |
| `A-B` | `ab` | `ab` | ← `A・B` と一致 |
| `むらの・たみ` | `むらのたみ` | `むらのたみ` | |
| `むらのたみ` | `むらのたみ` | `むらのたみ` | ← `むらの・たみ` と一致 |
| `ムラノタミ` | `ムラノタミ` | `むらのたみ` | ← `むらのたみ` と一致 |
| `ﾑﾗﾉﾀﾐ` | `ムラノタミ` | `むらのたみ` | ← 半角カナも一致 |
| `Ａ１Ｂ` | `a1b` | `a1b` | |
| `A1B` | `a1b` | `a1b` | ← 全角英数と一致 |
| `第一工房` | `第一工房` | `第1工房` | |
| `第1工房` | `第1工房` | `第1工房` | ← 漢数字と一致 |

##### 3.4A.3. 類似判定条件

各フォルダ名をメインキーとサブキーに分解（パース方法はセクション6.2参照）した上で、**類似判定用キー** を使用して以下のいずれかを満たす場合に「類似」と判定する。

| # | 条件名 | 説明 |
|---|---|---|
| 1 | メインキー一致 | 類似判定用メインキーが同一 |
| 2 | サブキー完全一致 | サブキーのセット（類似判定用キー）が完全一致（メインキーが異なっても） |
| 3 | キーセット一致 | メイン・サブの区別なく全キー集合（類似判定用キー）が同一（例: `A(B)` と `B(A)` を検出） |
| 4 | 包摂関係 | サブキーを持たないフォルダの類似判定用メインキーが、他フォルダのサブキー（類似判定用キー）に含まれる |

**グルーピング**: BFS（幅優先探索）による推移的クラスタリング。A≈B かつ B≈C であれば {A, B, C} を同一グループとする。

##### 3.4A.4. 統合UI

検出されたグループを一覧表示し、ユーザーが各グループの統合先を選択する。

##### グループ内の表示

各グループ内の候補には以下のラベルを付与し、種類を視覚的に区別する。

| ラベル | 意味 |
|---|---|
| (実フォルダ) | 出力フォルダ内に実在するサブフォルダ |
| (辞書) | 辞書に登録されている正式フォルダ名（実フォルダとして存在しない場合） |

辞書名が存在するグループでは、辞書名を推奨候補として強調表示する。

##### 表示例

```
グループ1:
  ○ むらの・たみ     (辞書) ← 推奨
  ○ むらのたみ       (実フォルダ)
  ○ ムラノタミ       (実フォルダ)
  ○ 統合しない

グループ2:
  ○ FolderA          (実フォルダ)
  ○ FolderA(FolderB) (辞書) ← 推奨
  ○ FolderB          (実フォルダ)
  ○ 統合しない
```

##### 統合先選択のルール

- ユーザーがグループ内の任意の候補を統合先として選択できる
- 「統合しない」を選択した場合、そのグループはスキップする
- 辞書名を統合先として選択した場合:
  - 統合先フォルダが存在しなければ新規作成する
  - 実フォルダから統合先へファイルを移動し、リネームと統合が同時に実現される

##### 3.4A.5. 統合処理

ユーザーが統合先を選択し実行を指示した場合:

1. 統合先フォルダが存在しない場合は新規作成する
2. 統合元フォルダ内の全ファイルを統合先フォルダへ移動する
   - ファイル名衝突時は `_1`, `_2` サフィックスで回避する（セクション10参照）
3. 空になった統合元フォルダを削除する
4. 1グループでエラーが発生しても他グループの処理は継続し、最後にまとめてエラーを報告する

##### 3.4A.6. 辞書の自動更新

統合処理の完了後、統合結果に基づいて辞書を自動更新する。

- 統合先フォルダに対して、グループ化ルール（セクション6.2）に従いキーを登録する
- 統合元フォルダ名から生成されるキーも統合先フォルダにマッピングする
  - 例: `FolderA` を `FolderB(FolderA)` に統合した場合、`normalize("FolderA")` → `FolderB(FolderA)` を辞書に追加
- 辞書更新後、辞書ファイルを保存する
- 辞書保存後、ドライランの自動再実行をトリガーする（F-05参照）

---

#### 3.4B. 少数ファイルフォルダ整理

ファイル数が少ないフォルダを検出し、ユーザーが定義したグループに統合する。
大量の小規模フォルダを整理・集約する際に使用する。

##### 3.4B.1. 検出

- ユーザーがしきい値（ファイル数）を指定する（例: 3件以下）
- 出力フォルダ内のサブフォルダを走査し、含まれるファイル数がしきい値以下のフォルダをリストアップする

##### 3.4B.2. グルーピングUI

検出されたフォルダの一覧を表示し、ユーザーが統合先グループを定義・割り当てる。

**操作フロー**:

1. しきい値を入力し「検出」を実行すると、対象フォルダ一覧が表示される
2. ユーザーは統合先グループを作成する（グループ名＝統合後のフォルダ名、編集可能）
3. 対象フォルダを選択し、任意のグループに割り当てる
4. どのグループにも割り当てないフォルダはそのまま残す（統合しない）
5. 割り当て完了後「実行」で統合を開始する

**UI構成**（類似フォルダ統合UIとの統一感を持たせる）:

```
しきい値: [3] 件以下  [検出]

─── 検出結果: 15件 ───

グループ1: [まとめA        ] (編集可能)
  ☑ FolderX    (2件)
  ☑ FolderY    (1件)

グループ2: [まとめB        ] (編集可能)
  ☑ FolderZ    (3件)

─── 未割り当て ───
  □ FolderW    (2件)
  □ FolderV    (1件)

[+ グループ追加]          [実行] [キャンセル]
```

**操作の詳細**:

| 操作 | 説明 |
|---|---|
| グループ追加 | 新しい統合先グループを作成する。グループ名（＝統合後のフォルダ名）を入力する |
| グループ名編集 | グループ名をインラインで編集できる。これが統合後の実フォルダ名になる |
| フォルダの割り当て | 未割り当てのフォルダを選択し、既存のグループにドラッグまたはボタンで割り当てる |
| 割り当て解除 | グループ内のフォルダを選択し、未割り当てに戻す |
| グループ削除 | グループを削除する。割り当て済みフォルダは未割り当てに戻る |

##### 3.4B.3. 統合処理

ユーザーが「実行」を指示した場合:

1. 各グループについて、グループ名のフォルダを出力先に作成する（存在しない場合）
2. グループに割り当てられた各フォルダ内の全ファイルを、グループ名フォルダへ移動する
   - ファイル名衝突時は `_1`, `_2` サフィックスで回避する（セクション10参照）
3. 空になった元フォルダを削除する
4. 1グループでエラーが発生しても他グループの処理は継続し、最後にまとめてエラーを報告する

##### 3.4B.4. 辞書の自動更新

統合処理の完了後、辞書を自動更新する。

- 各グループの統合先フォルダに対して、グループ化ルール（セクション6.2）に従いキーを登録する
- 統合元フォルダ名から生成されるキーも統合先フォルダにマッピングする
- 辞書更新後、辞書ファイルを保存する
- 辞書保存後、ドライランの自動再実行をトリガーする（F-05参照）

### F-05: ドライラン（プレビュー）

メインウィンドウ内のプレビューパネルに、分類結果のシミュレーションを常時表示する。
ファイル操作は行わず、分類ロジックの結果のみを算出・表示する。

#### 3.5.1. 自動実行トリガー

以下のイベント発生時にドライランを自動的に実行する。

| トリガー | タイミング | 実行条件 |
|---|---|---|
| アプリ起動時 | 設定読み込み完了後 | 入力フォルダが設定済み かつ フォルダが実在する |
| 入力フォルダ選択時 | フォルダ選択ダイアログで確定後 | 選択されたフォルダが実在する |
| 辞書更新時 | 辞書編集で「保存」実行後 / 辞書作成完了後 / 出力フォルダ整理完了後 | 入力フォルダが設定済み |
| スキャンオプション変更時 | 再帰スキャン ON/OFF 切替時 | 入力フォルダが設定済み |

#### 3.5.2. 実行制御

- **非同期実行**: バックエンドで非同期に処理し、UIをブロックしない
- **デバウンス**: 短時間に複数のトリガーが発生した場合、最後のトリガーのみを実行する
- **キャンセル**: 実行中に新たなトリガーが発生した場合、実行中の処理をキャンセルし最新のトリガーで再実行する
- **スキャン中表示**: 結果が返るまでプレビューパネルに「スキャン中...」を表示する

#### 3.5.3. 結果表示

メインウィンドウ内のプレビューパネルにテーブル形式で表示する。

| 列 | 内容 |
|---|---|
| ファイル名 | 対象ファイルの名前 |
| 分類先 | 辞書解決後のフォルダ名（クリックで編集可能） |
| 状態 | 分類可能 / 分類不可（タグなし） |

- **仮想スクロール**: TanStack Virtual の仮想スクロールを使用し、大量ファイル（1万件以上）でも軽量に描画する
- **サマリ表示**: テーブル上部に件数サマリ（分類可能 / 分類不可）を表示する
- **実行ボタン**: プレビュー結果を確認後、そのまま分類を実行できるボタンを配置する

#### 3.5.4. 分類先のインライン編集

プレビューの分類先セルをクリックすることで、分類先フォルダ名をインラインで編集できる。

- 編集はdestination単位でまとめて反映される（同じ分類先の全ファイルが一括変更）
- 編集済みの行は背景色（黄色）で視覚的に区別する
- ドライラン再実行時に編集状態はクリアされる

#### 3.5.5. 分類実行時の辞書更新

プレビューで分類先を編集した場合、分類実行時に辞書が自動更新される:

- **辞書未登録だったタグ（自動作成）の編集**:
  - タグ名 → 編集後のフォルダ名のマッピングを辞書に追加
  - 編集後のフォルダ名からグループ化ルール（セクション6.2）に従いキーを生成・登録
- **辞書にヒットしていたタグの編集**:
  - 編集前のフォルダ名に紐づく全キーの値を編集後のフォルダ名に更新
  - 編集前・編集後のフォルダ名の両方からキーを生成し、編集後のフォルダ名にマッピング

### F-06: 処理ログ

分類・統合・辞書同期の各操作について処理結果を記録し、ユーザーが確認できるようにする。

#### 保持方式

- **メモリ上に保持**: ログはアプリ起動中のみ保持し、ファイルには保存しない
- **アプリ終了時にクリア**: 再起動するとログはリセットされる

#### 記録内容

各レコードに以下の情報を含む:

| フィールド | 型 | 説明 |
|---|---|---|
| `timestamp` | string (ISO 8601) | 処理日時 |
| `operation` | string | 操作種別（`classify`, `merge`, `sync` 等） |
| `source` | string | 対象ファイルパス |
| `destination` | string | 分類先フォルダパス（該当時） |
| `action` | string | 実行内容（`copy`, `move`, `skip`, `error`） |
| `detail` | string | 補足情報（スキップ理由、エラー内容等） |

#### UI表示

- **別画面**で表示する（メインウィンドウからボタンまたはメニューで開く）
- 処理ログを時系列で一覧表示する
- 操作種別（成功/スキップ/エラー）による絞り込みが可能
- 処理完了時に、メインウィンドウ上で処理件数のサマリ（成功数・スキップ数・エラー数）を表示する

### F-07: 処理中断

分類処理の実行中に、ユーザーがUIから停止操作を行える。

- 実行中、停止ボタンを表示する
- 停止はファイル単位のグレースフル停止とする（処理中のファイルは最後まで完了させてから停止）
- 停止時点までに処理済みのファイルは元に戻さない（ロールバックしない）
- 停止後、ステータスに「中断」を表示し、処理済み件数を通知する

### F-08: 辞書フォーマットチェック

辞書JSONファイルの読み込み時に、内容の妥当性を検証する。

#### チェック項目

| # | チェック内容 | 不正時の動作 |
|---|---|---|
| 1 | JSONとしてパース可能か | エラーメッセージを表示し、辞書を空の状態で起動。ユーザーにファイル修正を促す |
| 2 | トップレベルがオブジェクト（`{}`）か | 同上 |
| 3 | すべての値が文字列型か | 不正な値を持つエントリを警告表示し、該当エントリをスキップして読み込む |
| 4 | キーが正規化ルール（セクション5）に準拠しているか | 非準拠のキーを警告表示。読み込みは継続する |

#### 設計意図

辞書ファイルはEXEと同じフォルダに配置されるプレーンなJSONファイルであり、ユーザーが外部テキストエディタやエクスプローラ経由で直接編集・コピーすることを想定する。そのため、手動編集によるフォーマット崩れを検出し、ユーザーに適切なフィードバックを返すことが重要である。

## 4. タグ名抽出仕様

ファイル名から分類用のタグ名を抽出する処理。

### 4.1. 抽出ルール

- ファイル名から**最初に出現する** `[` と `]` の間の文字列を取得する
- 前後の空白を除去する
- `[` と `]` が存在しないファイルは分類対象外とし、スキップする

### 4.2. セキュリティ対策（サニタイズ）

抽出された文字列はフォルダ名として使用されるため、以下のサニタイズを必ず実施する。

| # | 処理 | 理由 |
|---|---|---|
| 1 | `/`, `\`, `:` を削除 | パス区切り文字によるディレクトリトラバーサル防止 |
| 2 | `..` を繰り返し除去 | 親ディレクトリ参照の無効化 |
| 3 | 先頭の `.` を除去 | 隠しフォルダの作成防止 |
| 4 | サニタイズ後に空文字列になった場合はスキップ | 無効なタグの除外 |

## 5. 正規化（Normalization）仕様

辞書キーの生成に使用する文字列正規化処理。辞書のキーはすべてこの正規化を経た文字列とする。

### 5.1. 処理ステップ（順序厳守）

| Step | 処理 | 説明 |
|---|---|---|
| 1 | NFKC正規化 | Unicode互換分解＋標準合成。全角英数→半角 等 |
| 2 | 小文字化 | 英字をすべて小文字に統一 |
| 3 | 空白除去 | 全種類の空白文字を完全に除去 |
| 4 | 文字フィルタリング | 許可文字（5.2参照）のみを残し、それ以外を削除 |

### 5.2. 許可文字

- **英数字**: Unicode の `isAlphanumeric` 判定が真の文字（ひらがな・カタカナ・漢字・韓国語文字等を含む）
- **丸括弧**: `(`, `)`
- **カンマ**: `,`
- 上記以外のすべての文字は削除する

### 5.3. 正規化例

| 入力 | 出力 |
|---|---|
| `フォルダ 名A` | `フォルダ名a` |
| `Folder Name A` | `foldernamea` |
| `Folder(B)` | `folder(b)` |
| `！Full Width！` | `fullwidth` |
| `A, B` | `a,b` |

## 6. 辞書仕様

### 6.1. データ構造

JSON形式のファイルとして保存する。ユーザーが外部エディタで直接編集したり、エクスプローラでコピーして別環境に移行することを想定する。

```json
{
  "正規化キー": "分類先フォルダ名（実名）"
}
```

- **キー**: 正規化済み文字列（セクション5のルールに従う）
- **値**: 出力先に使用する実際のフォルダ名（文字列型）

### 6.2. グループ化ルール（Main(Sub) 構造）

フォルダ名が `Main(Sub1, Sub2)` という形式の場合、以下の全キーを同一の値に登録する。
これにより、サークル名でもメンバー個人名でも同一フォルダに分類される。

| 登録キー | 値 |
|---|---|
| `normalize("Main(Sub1, Sub2)")` — フルネーム | `Main(Sub1, Sub2)` |
| `normalize("Main")` — メインキー | `Main(Sub1, Sub2)` |
| `normalize("Sub1")` — サブキー1 | `Main(Sub1, Sub2)` |
| `normalize("Sub2")` — サブキー2 | `Main(Sub1, Sub2)` |

**パースルール**:
- 正規表現 `^(.*?)\((.*)\)$` でメイン部分と括弧内を分離する
- 括弧内はカンマ `,` で分割してサブキーとする
- 括弧がない場合はフォルダ名全体をメインキーとし、サブキーなしとする

### 6.3. 自動登録

分類実行時に辞書に未登録のキーが出現した場合:
- 抽出されたタグ名（Raw Folder Name）をそのまま分類先フォルダ名として使用する
- `normalize(Raw Folder Name)` → `Raw Folder Name` のマッピングを辞書に追加する
- 処理完了後に辞書ファイルを保存する

### 6.4. 辞書ファイルパス

- EXEと同じディレクトリ（アプリケーションルート）に配置する
- 設定ファイル（セクション7）の `dict_path` で相対パスを指定可能
- デフォルト: `folder_dictionary.json`

### 6.5. フォーマットチェック

辞書ファイル読み込み時にF-08のフォーマットチェックを実施する。

## 7. 設定仕様

### 7.1. 設定ファイル

- ファイル名: `settings.json`
- 配置場所: EXEと同じディレクトリ（アプリケーションルート）
- 形式: JSON

### 7.2. 設定項目

| キー | 型 | デフォルト | 説明 |
|---|---|---|---|
| `input_dir` | string | `""` | 入力フォルダパス |
| `output_dir` | string | `""` | 出力フォルダパス |
| `dict_path` | string | `"folder_dictionary.json"` | 辞書ファイルパス（相対パス） |

| `recursive_scan` | bool | `false` | `true`: 再帰スキャン / `false`: 直下のみ |

- 読み込み時はデフォルト値と深いマージ（deep merge）を行う。ファイルに存在しないキーはデフォルト値で補完する
- 分類実行時にUIの状態を設定ファイルに保存する

|---|---|
| ファイル分類 | 1万ファイル規模でも実用的な速度で処理できること |
| ドライラン | 1万ファイル規模で数百ms以内に完了すること（ファイル名列挙 + 文字列処理のみ） |
| 類似検出 | 1000フォルダ規模で実用的な速度で完了すること |
| UIの応答性 | 処理中もUIがフリーズしないこと |
| プレビューテーブル | 1万件以上のレコードを仮想スクロールで軽量に描画できること |

### 9.2. セキュリティ

- タグ名抽出時のパストラバーサル対策（セクション4.2）を必須とする
- ユーザー入力由来の文字列をファイルパスに使用する箇所では必ずサニタイズを実施する

### 9.3. 信頼性

| 項目 | 要件 |
|---|---|
| エラー通知 | ファイル操作のエラーはGUI上でユーザーに通知する |
| 部分エラー時の継続 | 1ファイルの処理失敗で全体が停止しない。エラーを記録して次のファイルに進む |
| データ整合性 | 辞書ファイルが破損・不正な場合、F-08のフォーマットチェックにより検出し、明確なエラーメッセージを表示する |

### 9.4. アーキテクチャ

| 項目 | 要件 |
|---|---|
| フロント/バックエンド分離 | ファイル操作・辞書管理・類似判定等のコアロジックはすべてRust（バックエンド）側に実装する。フロントエンドはUI表示と操作のみを担当する |
| コマンドインターフェース | フロントエンド↔バックエンド間の通信は Tauri の Command（invoke）を使用する |
| イベント通知 | 進捗通知など非同期のバックエンド→フロントエンド通知は Tauri のイベントシステムを使用する |
| テスト可能性 | コアロジックに対してユニットテストを作成可能な構造とする |

## 10. ファイル名衝突回避仕様

出力先に同名ファイルが既に存在する場合の処理。

- ファイル名の拡張子前に `_1`, `_2`, `_3` ... のサフィックスを付与する
- 例: `document.txt` → `document_1.txt` → `document_2.txt`
- 空き番号が見つかるまで連番をインクリメントする

## 11. アプリケーションルート仕様

設定ファイル・辞書ファイルの配置基準となるディレクトリの決定ルール。

| ファイル | 配置場所 |
|---|---|
| `settings.json` | アプリケーションルート |
| `folder_dictionary.json` | アプリケーションルート（`dict_path` で変更可能） |

- EXEファイルが存在するディレクトリをアプリケーションルートとする
- 開発時（`cargo tauri dev`）は、プロジェクトルートをアプリケーションルートとして扱う
- 辞書ファイルはユーザーがエクスプローラで自由にコピー・移動できるプレーンなJSONファイルである
